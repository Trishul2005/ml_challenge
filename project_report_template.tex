\section{Model Exploration}

% \textbf{Worth 1 point. No more than 1 page.} Describe the model(s) that you evaluated/explored.

The models that we explored were Naive Bayes, Decision Tree, Random Forest, K Nearest Neighbours, Multi Layer Perceptron, and combining an ensemble of models. Before evaluating the models we first pre-processed the data to ensure the features were consistent. The pre-processing included replacing categorical columns with one-hot-vectors, blank responses with empty strings, and only looking at English text. For text entry features, we used TF-IDF vectorization to give higher weights to words that appear often in a particular example but are less common in the entire dataset. Using TF-IDF was a natural decision since it helps the models make use of certain words to make a prediction. These pre-processed features were used as the initial input features so that we could compare the models more consistently.

For Naive Bayes, we explored the initial sklearn TF-IDF version and later experimented with bag-of-words. Since the data had free text word features, using Naive Bayes was a good fit for the data. In the bag-of-words exploration we built it using the three most indicative categories which were food, soundtrack, and feeling. This made Naive Bayes perform better, while keeping the model lightweight.

For the tree models, we looked at Decision Trees and Random Forest. The Decision Tree was used as a baseline using the initial feature representation without any extra tuning. This allowed us to identify that feature splits could be a useful method for prediction. With Random Forest, we used a more thought out feature choice by using the same three indicative categories from above, bag-of-words, and one hot vectors for categorical options. This representation consistently performed better than our initial input features.

We also briefly explored K Nearest Neighbours and Multi Layer Perceptron on the initial TF-IDF data. For KNN, we looked at different values of K, however the prediction results were far weaker than the earlier models so we did not continue. MLP was used to experiment with a neural network and we further explored with the same detailed input features as Random Forest. The results for MLP were good, however they were still behind Naive Bayes and Random Forest.

Finally, from the exploration we also tried an ensemble approach. This looked like a vote system where we ran Random Forest, Naive Bayes, and MLP together and then picked the most common prediction. Ties were broken by defaulting to Random Forest since that was our highest accuracy model. This model combination resulted in our best accuracy numbers, but the complexity to implement multiple models and within the size constraint of the project made us focus on only one model.

Overall, our main choice was to begin with a common shared choice for all models, then develop a more detailed input feature choice for models with the most promising initial accuracies. This helped us compare models early on, and focus on building stronger models such as Random Forest and Naive Bayes to better match the structure of the dataset.

When considering which model to use, we considered three main candidates: Na\"ive bayes, multi-layer perceptrons, and random forests. To evaluate which model to choose we looked at the three models average performance as well as peak performance over hundreds of different training-test splits. We did this because we noticed that there is a significant difference in validation accuracy depending on our split on untuned models, so evaluating on one split could bias one model over another. To maintain consistency, we kept evaluated the same split for each of the three models and found that random forests had the highest validation accuracy on avg and in its extrema before hyperparameter tuning. We also did a higher volume of random splits on each model without keeping them consistent as that was faster when we considering a single model and the results were consistent with when we homologated the splits between model families. So our evaluation metric was aimed at being as fair as possible while keeping iteration speed up.

Specifically we saw Na\"ive bayes had on average a 74\% validation accuracy with a 90\% training accuracy and MLPs having around 88\% validation accuracy with a 95\% training accuracy. These were averages over ~100-200 runs with identical train-validation-test splits for each model. We also tracked the peak performance of each model which followed the same trend. This gave us confidence that the random forest family would provide the best model we could train as it outpreformed Na\"ive bayes and MLPs in both mean and peak validation accuracy across all runs.

Once we went with Random forest, we considered 2 hyperparameters to tune: tree depth and number of trees. We went this these because they determine how tightly each tree can approximate a function and how many such trees we would be averaging across. We did binary search on two axes, in each iteration we took one hyperparameter and shifted it, leaving the other constant. This was to reduce ambiguity of the exact cause of decrease/increase of performance. We evaluated performance by a model's mean validation error across many splits. Once again since split choice was a large source of variance we ran each test 10-20 times across different splits allowing the random forest to be trained 100 times per choice of split to allow for variance due to ties to average out.

From these data points you can see that (250, 150) is a local minimum in the hyperparameter space. We neglected hunting beyond increments of 50 because that would result in diminishing returns and a lot more effort, especially when we have a model with 94\% validation accuracy.

Our submitted \texttt{pred.py} implements this final choice directly. The model is a random forest with \texttt{n\_estimators = 250}, \texttt{max\_depth = 150}, \texttt{class\_weight = balanced\_subsample}, and \texttt{max\_features = sqrt}. Rather than retraining at prediction time, \texttt{pred.py} loads a pre-exported forest from \texttt{final\_forest.npz}. That file stores the class names, the list of structured feature names, the text vocabulary, and the arrays for every decision tree in the forest. The inference script then manually walks each tree and sums the class votes, so the final submitted predictor does not depend on \texttt{sklearn} at runtime.

The feature representation in \texttt{pred.py} matches the feature engineering used in our development code (\texttt{model\_selection.py} / \texttt{predgpt.py}). For the structured part, we convert the three numeric survey responses into clipped numeric features, add missing-value indicator features, parse the price response and also include \texttt{log(1 + price)}, encode the four Likert-scale questions as integers from 1 to 5 with corresponding missing indicators, and one-hot encode each option selected in the three multi-select categorical questions. We also add simple interaction features: intensity $\times$ colours, intensity $\times$ objects, and colours $\times$ objects. For the text part, we concatenate the three free-response fields, tokenize with the same word pattern used during development, and count both unigrams and bigrams. In training, these text features were filtered with \texttt{min\_df = 2} and capped at \texttt{max\_features = 8000}; the exported vocabulary is then reused exactly at test time.

So, the final model described in this section and the final \texttt{pred.py} file are consistent: both use the same hybrid structured-plus-bag-of-words representation, and both correspond to a 250-tree random forest of depth 150 trained on the labeled dataset and then exported for lightweight inference.

\section{Prediction}

We estimate that our final model will achieve a test accuracy of \textbf{89.5\%}. This estimate is based on repeated runs of our \texttt{pred.py} pipeline. Across the 11 runs we recorded, the training accuracy was consistently 97.63\%, the validation accuracy averaged 92.49\%, and the test accuracy averaged 89.36\%. Since the observed test performance is slightly below the validation performance on average, we do not expect the model to match its validation score exactly on unseen data.

Our estimate of 89.5\% is justified by the overall pattern in these experiments. The average gap between validation and test accuracy is about 3.13 percentage points, while the gap between training and test accuracy is about 8.26 percentage points. This suggests that the model fits the training set very well but still shows some overfitting, so a test estimate should be lower than both the training and validation accuracies. In addition, most of the observed test accuracies cluster near 89\%--90\%, even though the best individual run reached 90.51\%. Using the single best run would likely be too optimistic, since model selection based on validation performance can overestimate future generalization. For that reason, a point estimate close to the empirical average test accuracy is the most defensible prediction.
